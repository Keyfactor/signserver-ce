    Larger Exercise: Text Signer and Validator

    In this larger exercise we will create a mostly complete signer and validator for a made up format that we call "Text Signature".

    The purpose is to more easily demonstrate the SignServer API for implementing signers and validators without the complexity of a real-world example. The format described here is not intended for production use. In fact, in general it is not advisable to invent your own format but instead as far as possible first consider using a well-established specification.

    Specification

    The ’Text Signature’ format is:

    • a totally made up format

    • inspired by PKCS#7/CMS for the use of signed attributes

    • inspired by PEM and OpenPGP for the ASCII armoring

    • only supporting a single signer so no counter-signatures nor parallel signatures

    • not supporting unsigned attributes so no time-stamping and embedded revocation information

    • most likely with weaknesses and/or other types of limitations

    Format

    -----BEGIN SIGNED TEXT-----
    This is the original document.
    It can contain multiple lines.
    -----BEGIN TEXT SIGNATURE-----
    signature=...base64 encoded signature...
    location=Stockholm
    signingTime=Fri Mar 20 10:55:01 CET 2015
    certificates=...PEM encoded certificates...
    contentDigest=...base64 encoded digest...
    contentDigestAlgorithm=SHA256
    signatureAlgorithm=SHA256withRSA
    -----END TEXT SIGNATURE-----

    Where:

    • The text between the BEGIN SIGNED TEXT and BEGIN TEXT SIGNATURE headers (ie the content) is the original document but with a line endings using LF instead of CR or CRLF.

    • The text between BEGIN TEXT SIGNATURE and END TEXT SIGNATURE is using the Java Properties file format.

    • contentDigest is the hash of the content using the algorithm specified in contentDigestAlgorithm.

    • contentDigestAlgorithm is the algorithm to use to hash the content.

    • Certificates is the PEM encoded signer certificate followed by any intermediate CA certificates.

    • signature is the output from the signing function. Input to the signature function is each other attributes with the name followed by a equals sign followed by the value followed by a newline. Ie: attributeName + "=" + attributeValue + LF. The attributes are to be taken sorted on the attributeName.

    • signatureAlgorithm is the algorithm to use for the signature.

    • Other attributes could be included as well.

    12.10.2 Exercise: Implementing TextSigner

    In this exercise we will implement a signer using the previously specified format. The following worker properties will be configurable:

    Worker Property

    Description

    DIGESTALGORITHM

    Algorithm for hashing the content (Optional, default: "SHA-256")

    SIGNATUREALGORITHM

    Algorithm for signing the attributes (Optional, default: "SHA256withRSA")

    LOCATION

    Attribute to indicate the geographical location (Optional, default: not included)

    ALLOW_LOCATION_OVERIDE

    True if the location attribute could be taken from the request (Optional, default: "False")

    This signer may accept the following property within the request as a request meta data:

    Request Metadata Property

    Description

    LOCATION

    To use as the location attributes (Optional, requires ALLOW_LOCATION_OVERRIDE to be configured)

    To speed up the development we will also this time base the worker on a skeleton implementation from the SignServer-Module-Sample project.

    We will also for simplicity place the implementation in the existing SignServer-Module-XAdES project even though it should probably be placed in a different project.

    1. Make sure the SignServer-Module-XAdES project is opened in the IDE.

    2. Right click on the SignServer-Module-XAdES project and choose New → Java Package...

    3. Write "org.signserver.training" and press Finish.

    4. Make sure the SignServer-Module-Sample project is opened in the IDE.

    5. Find and copy the file SkeletonSigner.java from the Sample project to the new org.signserver.training package giving it the new Name "TextSigner". Click Refactor button.

    6. Open TextSigner.java in the org.signserver.training package and verify that it starts with:

      package org.signserver.training;
      ...
      public class TextSigner extends BaseSigner {

    7. We start by adding constants for the worker and request properties:

      // Worker properties
      public static final String PROPERTY_SIGNATUREALGORITHM
      = "SIGNATUREALGORITHM";
      public static final String PROPERTY_DIGESTALGORITHM
      = "DIGESTALGORITHM";
      public static final String PROPERTY_LOCATION
      = "LOCATION";
      public static final String PROPERTY_ALLOW_LOCATION_OVERRIDE
      = "ALLOW_LOCATION_OVERRIDE";

    8. All worker properties are optional and we add some constants with the default values as well:

      // Default values
      private static final String DEFAULT_SIGNATUREALGORITHM
      = "SHA256withRSA";
      private static final String DEFAULT_DIGESTALGORITHM
      = "SHA256";
      private static final String DEFAULT_LOCATION
      = null;
      private static final boolean DEFAULT_ALLOW_LOCATION_OVERRIDE
      = false;

    9. When we are at it, we also fill in the content-type for the request and response:

      // Content types
      private static final String REQUEST_CONTENT_TYPE = "text/plain";
      private static final String RESPONSE_CONTENT_TYPE = "text/plain";

    10. And also adding a constant for logging the location the client specified:

      // Log fields
      public static final String LOG_REQUESTED_LOCATION = "REQUESTED_LOCATION";

    11. Now we need some fields to store the configuration values in after they have been parsed:

      // Configuration values
      private String digestAlgorithm;
      private String signatureAlgorithm;
      private String location;
      private boolean allowLocationOverride;

    12. Time to implement the init() method and parse the configuration. Add the parsing of the DIGESTALGORITHM property:

      // Read properties
      // Optional property DIGESTALGORITHM
      digestAlgorithm = config.getProperty(PROPERTY_DIGESTALGORITHM);
      if (digestAlgorithm == null || digestAlgorithm.trim().isEmpty()) {
      digestAlgorithm = DEFAULT_DIGESTALGORITHM;
      }

    13. And the LOCATION property:

      // Optional property LOCATION
      location = config.getProperty(PROPERTY_LOCATION);
      if (location == null || location.trim().isEmpty()) {
      location = DEFAULT_LOCATION;
      }

    14. And finally for the ALLOW_LOCATION_OVERRIDE property:

      // Optional property ALLOW_LOCATION_OVERRIDE
      final String value = config.getProperty(
      PROPERTY_ALLOW_LOCATION_OVERRIDE);
      if (value == null || value.trim().isEmpty()) {
      allowLocationOverride = DEFAULT_ALLOW_LOCATION_OVERRIDE;
      } else if (Boolean.TRUE.toString().equalsIgnoreCase(value.trim())) {
      allowLocationOverride = true;
      } else if (Boolean.FALSE.toString().equalsIgnoreCase(value.trim())) {
      allowLocationOverride = false;
      } else {
      configErrors.add("Incorrect value for property "
      + PROPERTY_ALLOW_LOCATION_OVERRIDE);
      }

      Note that any configuration error that will prevent our worker from functioning is added to the configErrors list. We are also on purpose doing a bit of extra work to really make sure we report an error in case the property does not have a good value.

    15. Now it’s time to do the actual implementation in the processData() method. After getting the data from the request we will reformat it so that it matches the specification, ie. the line endings are changed to only use line feeds (LF). For that we use a helper method canonicalize(). We also obtain the location from the request meta data:

      // Get the data from request
      final ReadableData requestData = request.getRequestData();
      final WritableData responseData = request.getResponseData();
      final byte[] data = requestData.getAsByteArray();
      final String text = canonicalize(data);
      final String requestedLocation = RequestMetadata.getInstance(
      requestContext).get(PROPERTY_LOCATION);

    16. And the canonicalize() method is like this:

      public static String canonicalize(byte[] data) throws IOException {
      final StringBuilder sb = new StringBuilder();
      final BufferedReader in = new BufferedReader(new InputStreamReader(
      new ByteArrayInputStream(data), StandardCharsets.UTF_8));
      String line;
      while ((line = in.readLine()) != null) {
      sb.append(line).append("\n");
      }
      return sb.toString();
      }

    17. Further down in the processData() method, right after the "Produce the results, ie doing the work" but before Certificate handling we insert some code to create the properties/attributes for contentDigest, contentDigestAlgorithm and signingTime:

      // Produce the result, ie doing the work
      Properties attributes = new Properties();
      // Content digest
      final MessageDigest md = MessageDigest.getInstance(digestAlgorithm);
      final byte[] contentDigest = md.digest(
      text.getBytes(StandardCharsets.UTF_8));
      attributes.setProperty("contentDigest",
      Base64.toBase64String(contentDigest));
      attributes.setProperty("contentDigestAlgorithm", digestAlgorithm);
      // time
      attributes.setProperty("signingTime", new Date().toString());

    18. Then there is the location attribute which is a bit special as the value can, if allowed, come from the request:

      final String effectiveLocation;
      if (requestedLocation == null) {
      // No location in request so use the configured one
      effectiveLocation = location;
      } else {
      // Location in request
      if (allowLocationOverride) {
      effectiveLocation = requestedLocation;
      } else {
      // Client error
      throw new IllegalRequestException("Requesting "
      + PROPERTY_LOCATION + " not allowed.");
      }
      }
      // Set the location value if we have one
      if (effectiveLocation != null) {
      attributes.setProperty("location", effectiveLocation);
      }

    19. Then the last attribute (before the actual signature) is the one holding the certificates in PEM format. To construct this we use the PEMWriter:

      // certifiates
      ByteArrayOutputStream bout0 = new ByteArrayOutputStream();
      PEMWriter pemWriter = new PEMWriter(
      new OutputStreamWriter(bout0));
      for (Certificate cert : includedCertificates(
      getSigningCertificateChain(cryptoInstance))) {
      pemWriter.writeObject(cert);
      }
      pemWriter.flush();
      attributes.setProperty("certificates", bout0.toString(StandardCharsets.UTF_8.name()));

    20. Now it is time to calculate the signature value. In the skeleton the data is directly passed to the signature object like this: signature.update(data) but we feed each attribute one at the time to the signing function. For the signature verification to be deterministic it is specified that the attributes should be taken in alphabetical order so we will sort the attribute names first.

      At the following before the line with signature.sign(), replacing the "Feed the data to be signed" while-loop:

      final String[] attributeNames = attributes.stringPropertyNames()
      .toArray(new String[0]);
      // Sort as specified
      Arrays.sort(attributeNames);
      for (String attr : attributeNames) {
      String attribute
      = attr + "=" + attributes.getProperty(attr) + "\n";
      signature.update(attribute.getBytes(StandardCharsets.UTF_8));
      }
      // Get the final signature

    21. Then finally we will add the signature and render the finally result. For that we will use a helper method renderProperties for convenience.
      Replace the current "final String result = " with the following:

      // Render the result
      final StringBuilder sb = new StringBuilder();
      sb.append("----- BEGIN SIGNED TEXT -----\n");
      sb.append(text);
      sb.append("----- BEGIN TEXT SIGNATURE -----\n");
      sb.append(renderProperties(attributes));
      sb.append("----- END TEXT SIGNATURE -----\n");
      String result = sb.toString();

    22. The convenience method renderProperties is defined like this:

      private String renderProperties(Properties properties)
      throws IOException {
      ByteArrayOutputStream bout = new ByteArrayOutputStream();
      properties.store(bout, null);
      String s = bout.toString(StandardCharsets.UTF_8.name());
      final StringBuilder sb = new StringBuilder();
      BufferedReader sr = new BufferedReader(new StringReader(s));
      sr.readLine(); // Skipping first line
      String line;
      while ((line = sr.readLine()) != null) {
      line = line.replace("\\n", "\\n\\\n");
      // Check if (the new lines) need wrapping
      BufferedReader sr2 = new BufferedReader(
      new StringReader(line));
      String line3;
      while ((line3 = sr2.readLine()) != null) {
      while (line3.length() > 80) {
      String line2 = line3.substring(0, 80);
      sb.append(line2).append("\\\n");
      line3 = line3.substring(80, line3.length());
      }
      sb.append(line3).append("\n");
      }
      sb.append("\n");
      }
      return sb.toString();
      }

    23. Build and deploy SignServer:

      mvn install -DskipTests && bin/ant deploy-ear

      Make sure to check that the compilation and deployment finished with success otherwise go back and resolve the issues.

    24. Now to test the worker we need to configure it. Navigate to the Admin Web.

    25. Go to Workers → Add...

    26. This time as we haven’t created any sample file for our worker we will not create it from file. Instead choose By Properties.

    27. Fill in:
      Name: "MyTextSigner"
      Implementation Class: "org.signserver.training.TextSigner"

    28. Click button Next.

    29. Click button Apply and the worker is added but is listed as OFFLINE.

    30. Clicking on the worker name to get to the Status Summary page shows an error message saying that the worker is

      missing a crypto token and certificates.

    31. Click on Configuration tab and add a property called CRYPTOTOKEN with the value "CryptoTokenP12A" as well as a property DEFAULTKEY with the value "Signer 2". Now the worker should be ONLINE. Alternatively the "CryptoTokenP11A" could be chosen to use the HSM but in that case you will also need to generate a key-pair, create certificate request and get and install the certificates from the CA.

    32. Time to test the worker. Go to the client web page and the Generic Signing and Validation Demo page

    33. Fill in Worker name: MyTextSigner.

    34. Fill in some text as Data.

    35. Click Submit and you should get back a signature similar in form as the example.

    12.10.3 Exercise: Implementing TextValidator

    In this next exercise we will create the validation part for signatures created in the previous one.

    The only configurable option will be a list of all trusted CA certificates.

    To speed up the development we will also this time base the worker on a skeleton implementation from the SignServer-Module-Sample project. We will also for simplicity place the implementation in the existing SignServer-Module-XAdES project even though it should probably be placed in a different project/module.

    1. Make sure the SignServer-Module-XAdES and SignServer-Module-Sample projects are opened in the IDE.

    2. Expand the Source Packages under the SignServer-Module-XAdES project and make sure there is a package "org.signserver.training". If not create the package by right clicking and choose New → Java Package....

    3. Find and copy the file SkeletonValidator.java from the Sample project to the new org.signserver.training package giving it the new Name "TextValidator". Click Refactor button.

    4. Open TextValidator.java in the org.signserver.training package and verify that it starts with:

      package org.signserver.training;
      ...
      public class TextValidator extends BaseValidator {

    5. We start by adding constants for the worker property:

      // Worker properties
      public static final String PROPERTY_TRUSTANCHORS = "TRUSTANCHORS";

    6. We provide some fields to include in the worker log:

      // Log fields
      public static final String LOG_DIGEST_VALID = "DIGEST_VALID";
      public static final String LOG_SIGNATURE_VALID = "SIGNATURE_VALID";
      public static final String LOG_CERTIFICATE_VALID = "CERTIFICATE_VALID";

    7. Now we need some fields to store the configuration value in after it has been parsed:

      // Configuration values
      private final Set<TrustAnchor> trustAnchors = new HashSet<>();

    8. Time to implement the init() method and parse the configuration. Add the parsing of the TRUSTANCHORS property:

      // Read properties
      // Required property TRUSTANCHORS
      try {
      final String value = config.getProperty(PROPERTY_TRUSTANCHORS);
      if (value == null) {
      configErrors.add("Missing required property "
      + PROPERTY_TRUSTANCHORS);
      } else {
      final Collection<Certificate> certs = CertTools.getCertsFromPEM(
      new ByteArrayInputStream(value.getBytes(StandardCharsets.UTF_8)));
      if (certs.isEmpty()) {
      configErrors.add("Property " + PROPERTY_TRUSTANCHORS
      + " must not be empty");
      } else {
      // Add each certificate
      for (Certificate cert : certs) {
      if (cert instanceof X509Certificate) {
      trustAnchors.add(new TrustAnchor(
      (X509Certificate) cert, null));
      }
      }
      }
      }
      } catch (CertificateException ex) {
      configErrors.add("Property " + PROPERTY_TRUSTANCHORS
      + " caused error: " + ex.getMessage());
      }

    9. Now it’s time to do the actual implementation in the processData() method. To do that we need to cast the type of the request to a DocumentValidationRequest. Do that by replacing the following:

      final IValidationRequest request = (IValidationRequest) signRequest;
      if (!(request.getRequestData() instanceof byte[])) {
      throw new IllegalRequestException(
      "Unexpected request data type");
      }

      Instead write:

      final DocumentValidationRequest request = (DocumentValidationRequest) signRequest;

    10. After getting the data from the request we will start parsing out the original text from the signature format. Replace the "final byte[] data =" with:

      // Get the data from request
      final byte[] data = request.getRequestData().getAsByteArray();
      final BufferedReader in = new BufferedReader(new InputStreamReader(
      new ByteArrayInputStream(data), StandardCharsets.UTF_8));
      // First line should be header
      String line = in.readLine();
      if (line == null || !line.equals("----- BEGIN SIGNED TEXT -----")) {
      throw new IllegalRequestException(
      "Incorrect document, expected signed text header");
      }
      // Read the text, use "\n" as line ending as specified by TextSigner
      final StringBuilder sb = new StringBuilder();
      while ((line = in.readLine()) != null
      && !line.equals("----- BEGIN TEXT SIGNATURE -----")) {
      sb.append(line).append("\n");
      }
      if (line == null
      || !line.equals("----- BEGIN TEXT SIGNATURE -----")) {
      throw new IllegalRequestException(
      "Incorrect document, expected text signature header");
      }
      final String text = sb.toString();

    11. Next we parse out the signature (attributes) section into Properties:

      // Read the signature section
      final StringBuilder sb2 = new StringBuilder();
      while ((line = in.readLine()) != null
      && !line.equals("----- END TEXT SIGNATURE -----")) {
      sb2.append(line).append("\n");
      }
      if (line == null
      || !line.equals("----- END TEXT SIGNATURE -----")) {
      throw new IllegalRequestException(
      "Incorrect document, expected text signature footer");
      }
      final String signatureSection = sb2.toString();
      final Properties attributes = new Properties();
      attributes.load(new StringReader(signatureSection));
      final String[] attributeNames = attributes.stringPropertyNames()
      .toArray(new String[0]);
      // Sort as specified
      Arrays.sort(attributeNames);

    12. Now we can extract the content digest and check that it matches the digest over the text. Replace the "// Check signature" section with:

      // Check the digest over the text
      boolean validDigest;
      try {
      final String digestAlgorithm
      = attributes.getProperty("contentDigestAlgorithm");
      final byte[] digest = Base64.decode(attributes.getProperty(
      "contentDigest"));
      final MessageDigest md
      = MessageDigest.getInstance(digestAlgorithm);
      final byte[] actualDigest = md.digest(text.getBytes(StandardCharsets.UTF_8));
      validDigest = Arrays.equals(digest, actualDigest);
      } catch (NoSuchAlgorithmException ex) {
      validDigest = false;
      LOG.info("Message digest algorithm not supported: "
      + ex.getMessage());
      }

    13. To verify the signature we need to first get the signer certificate from the list of certificates.
      Replace the "Certificate cert = null;" with the following:

      // Check the signature over the attributes
      final List<Certificate> certificates
      = CertTools.getCertsFromPEM(new ByteArrayInputStream(
      attributes.getProperty("certificates")
      .getBytes(StandardCharsets.UTF_8)));
      final Certificate signerCertificate = certificates.get(0);

    14. Next step is to verify the signature. Replace the "boolean validSignature = false;" with the following:

      boolean validSignature;
      try {
      final String signatureAlgorithm
      = attributes.getProperty("signatureAlgorithm");
      final Signature signature
      = Signature.getInstance(signatureAlgorithm);
      signature.initVerify(certificates.get(0));
      for (String attr : attributeNames) {
      if (!"signature".equals(attr)) {
      // Construct the attributes string as specified
      // before hashing
      final String attribute = attr + "="
      + attributes.getProperty(attr) + "\n";
      signature.update(attribute.getBytes(StandardCharsets.UTF_8));
      }
      }
      final byte[] signatureBytes
      = Base64.decode(attributes.getProperty("signature"));
      validSignature = signature.verify(signatureBytes);
      } catch (Exception ex) {
      validSignature = false;
      }

      Note in the above code we are catching the raw Exception type for readability in the training. Normally this is considered bad practice and in a real implementation it could be replaced with multiple catch clauses for the different types of exceptions.

    15. Next we need to also validate the signer certificate and that it chains up to one of the configured trust anchors. To help with this we have the helper method validate().
      Replace the statements directly after "Check certificate and chain" with the following:

      // Check certificate and chain
      // Validate the signature certificate
      final Validation v
      = validate(signerCertificate, certificates, trustAnchors);
      final boolean validCertificate = v.getStatus() == Status.VALID;

    16. Now we have individual verified three parts of the format and stored the result in three boolean variables:
      a) validDigest: Hash over the text
      b) validSignature: Signature over the attribute
      c) validCertificate: Certificate valid and chaining up to a trusted certificate

    17. However, in the skeleton only the result from the signature and certificate are considered so we need to alter the return-statement so that it contains "validDigest" as well:

      // Return the response
      return new DocumentValidationResponse(request.getRequestID(),
      validDigest && validSignature && validCertificate,
      new CertificateValidationResponse(v, null));

    18. Let’s also log this individual values so it is possible to see in the log which part that failed validation:

      LogMap.getInstance(requestContext).put(LOG_DIGEST_VALID,
      String.valueOf(validDigest));
      LogMap.getInstance(requestContext).put(LOG_SIGNATURE_VALID,
      String.valueOf(validSignature));
      LogMap.getInstance(requestContext).put(LOG_CERTIFICATE_VALID,
      String.valueOf(validCertificate));

    19. That’s all. Now let’s build and deploy:

      mvn install -DskipTests && bin/ant deploy-ear

    20. Make sure to check that the compilation and deployment finished with success otherwise go back and resolve the issues.

    21. Now to test the worker we need to configure it. Navigate to the Admin Web.

    22. Go to Workers → Add...

    23. This time as we haven’t created any sample file for our worker we will not create it from file. Instead choose By Properties.

    24. Fill in:
      Name: "MyTextValidator"
      Implementation Class: "org.signserver.training.TextValidator"

    25. Click button Next.

    26. Click button Apply and the worker is added but is listed as OFFLINE.

    27. Clicking on the worker name to get to the Status Summary page shows an error message saying that the worker is

      missing the TRUSTANCHORS property.

    28. The TRUSTANCHORS property should contain the PEM encoded CA certificate used by the TextSigner. If the TextSigner is using CryptoTokenP12A then the file /opt/signserver/res/test/dss10/DSSRootCA10_cacert.pem is the certificate to use. Otherwise, if CryptoTokenP11A was used the certificates was manually installed and you should have the certificate chain file available to copy the last CA certificate from.

    29. Time to test the worker. Go to the client web page and the Generic Signing and Validation Demo page.

    30. Fill in Worker name: MyTextValidator.

    31. Copy in the output from a previous run of the TextSigner in the Data field or alternatively use the file upload form instead.

    32. Change Process type to Validate document.

    33. Click Submit and you should get back a response saying "VALID". If not check the output in the server log to see what failed.
