    Peer Systems

    Note: This is a SignServer Enterprise Edition (EE) feature.

    An SignServe instance can be setup to be the target of remote operations from an EJBCA instance. Connections are made using dual authenticated HTTPS. This is similar to how you use a client certificate to authenticate using the Admin GUI or Admin Web in order to manage a SignServer instance, but in this case the configured administrator is another EJBCA instance.

    Normally the instance with higher security requirements (e.g. an EJBCA acting as CA) initiates connections to a system with lower security requirements (e.g. a SignServer instance but it can also be an EJBCA acting as VA or RA). Currently, SignServer only implements support for incoming peer connections.

    Peer Systems for Certificate Renewal

    SignServer can be configured to expose the worker's keys as remote key bindings in EJBCA and that way allow the EJBCA administrators to issue new certificates to workers in SignServer directly from within the EJBCA interface and without incoming network connections to the CA.

    See the Operations Guide page Certificate Renewals using Peer Systems for step-by step instructions on how to typically set up the systems for certificate renewal using peer systems.

    Alternatively see the Renewal Worker which instead of Peer Systems uses the EJBCA web services interface but in that case connects in to the CA.

    Mapping SignServer Workers to Remote Key Bindings in EJBCA

    EJBCA has the concept called Internal Key Bindings and when those are accessed over peer systems those are seen as Remote Key Bindings. SignServer does not explicitly have Internal Key Bindings but a worker having a key assigned to it could be seen from the EJBCA side as a Remote Key Binding. How that is done will be described below.

    Listed Key Bindings

    All workers of worker type PROCESSABLE can be exposed as a remote key binding to EJBCA over peers if they have a crypto token either internally or by referencing one using the CRYPTOTOKEN worker property.

    By setting the worker property PEERS_VISIBLE=true the worker will be considered when returning the list of key bindings to EJBCA.

    Certificate Fingerprint

    After a worker has been renewed over peers, the worker will have the property PEERS_ISSUED=true indicating that the current certificate is issued by the remote peer. This will cause SignServer to include the certificate fingerprint when exposing the remote internal key binding. The consequence on the EJBCA side is that if it was the issuer of the certificate the web GUI will display a link to the certificate and match it to an end entity but if that certificate is not available in the EJBCA database, it will not display the remote internal key binding at all as it is assumed the certificate is then managed by an other CA. For this reason the PEERS_ISSUED property is not set for new workers (or set to false) so that EJBCA will list it and the first certificate can be issued.

    Offline Crypto Token

    If the crypto token is offline or the configured key-pair is not available the remote key pair can be showed as disabled in EJBCA and Renew or issuance buttons could be disabled. In that case first make sure to generate and configure the key-pair at the SignServer side.

    Support for EJBCA Before Versions 6.9

    Before EJBCA 6.9 when renewing a certificate only the end entity certificate is received and not any CA certificates. One would then have to manually append the CA certificates to the SIGNERCERTIFICATECHAIN property of the worker after each renewal. But in order to avoid having to this every time manually the worker property PEERS_KEEPCHAIN can be set to true. In this mode only the end entity certificate is replaced and the CA certificates remain. This works assuming that the CA certificates has been added before and that the CA certificates that should be used with the new certificate is the same as before. If the issuing CA changes then the CA certificate(s) has to be manually updated.

    Authentication/Authorization

    The incoming peer (i.e. the CA) needs to authenticate towards the application server using a TLS client certificate.

    Further on, in order for the peer to be allowed to list key bindings and to perform operations, the certificate must be authorized to the Peer System role. For managing authorizations see the Administrators Page (for Admin Web) or the wspeersystems command in the Administration CLI.

    Note that in the Admin Web it is possible to see incoming (i.e. ping) requests and easily add an authorization rule for it.

    Global Configuration Properties

    Property

    Description

    PEERS_INCOMING_ENABLED

    Set to true to allow incoming peer connections.

    Note: in Admin Web there is a check box for this on the Administrator Page.

    Worker Properties

    The following properties can be set on the workers that should be exposed as key binding to peer system(s).

    Property

    Description

    PEERS_VISIBLE

    Set to "true" to make the worker exposed as a internal key binding to all authorized peer systems.

    Default: "false".

    PEERS_ISSUED

    Indicates that the current key has been issued by the peer system.

    With this property set to "true", SignServer will include the fingerprint of the current certificate and EJBCA will only display the remote key binding if it has the issuer of the certificate. For that reason, before the worker has got its first certificate from the peer, this value should be "false" so that the key binding will be listed on the EJBCA side. Then after certificate has been issued, this property will automagically be changed to "true".

    Note: This property should normally not have to be changed manually.

    Default: "false".

    PEERS_KEEPCHAIN

    If the EJBCA version is before 6.9 it will only provide the end entity certificate without the chain. In that case setting this property to "true" will make the current CA certificates in the chain stay and only the end entity certificate is replaced when renewing. This way the CA certificates will only have to be added manually the first time (assuming the CA certificates does not change).

    DISABLED

    Disabling the worker by setting this to false will show the key binding as disabled to EJBCA.

    Default: "false".

    SIGNATUREALGORITHM

    The value of this property will be used as signature algorithm for signing the certificate signing request (CSR/PKCS#10) sent to the CA as part of the renewal.

    Default: "SHA256withRSA".

    KEYALG

    The value of this property will be used as key algorithm when generating the next key-pair.

    Default: "RSA".

    KEYSPEC

    The value of this property will be used as key specification (i.e. bit length for RSA or curve name for ECDSA) when generating the next key-pair.
    Default: "2048".

    REQUESTDN

    Subject DN to include in the certificate signing request (CSR/PKCS#10) sent to the CA as port of the renewal.

    Default: CN=<key alias>.
